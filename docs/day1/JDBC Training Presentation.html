What is JDBC?

  • Java Database Connectivity - Standard Java API for database access
  • Provides a set of Java classes and interfaces for database interaction
  • Database-independent connectivity to relational databases
  • Part of Java standard library (java.sql and javax.sql packages)
  • Uses JDBC Drivers to communicate with different databases

Why JDBC?

  • Write once, use with any relational database (driver pluggable)
  • Standardized API for all Java developers
  • Low-level API for fine-grained control over database operations
  • Foundation for higher-level frameworks (Hibernate, JPA, Spring Data)
  • Essential for understanding database connectivity in Java
  • Supports complex transactions and batch operations

Four Main JDBC Components

  • JDBC API - Java interfaces and classes (java.sql package)
  • JDBC Driver Manager - Manages database drivers and connections
  • JDBC Test Suite - Ensures compliance with JDBC standards
  • JDBC Drivers - Type 1, 2, 3, 4 driver implementations

JDBC Driver Types

  • Type 1: JDBC-ODBC Bridge - Uses ODBC (legacy, deprecated)
  • Type 2: Native API - Partially written in native code
  • Type 3: Network Protocol - Pure Java, middleware-based
  • Type 4: Thin Driver - Pure Java, direct database connection (RECOMMENDED)

Two-Tier Architecture

Java Application ↔ JDBC Driver ↔ Database

  • Direct communication between application and database
  • User commands sent directly to database
  • Results returned directly to application
  • Suitable for client-side applications and desktop software

Three-Tier Architecture

Client ↔ Middle Tier ↔ Database

  • Client-side application, middle-tier server, backend database
  • Middle tier handles business logic and database interactions
  • Suitable for web applications and enterprise systems
  • Better scalability and security

Core JDBC Interfaces

  • DriverManager - Manages JDBC drivers and creates connections
  • Connection - Represents a database connection
  • Statement - Execute SQL queries and updates
  • PreparedStatement - Precompiled SQL with parameters
  • CallableStatement - Execute stored procedures
  • ResultSet - Retrieve query results row by row

Loading JDBC Driver

Class.forName("com.mysql.cj.jdbc.Driver");

For different databases:

  • MySQL: com.mysql.cj.jdbc.Driver
  • PostgreSQL: org.postgresql.Driver
  • Oracle: oracle.jdbc.driver.OracleDriver
  • SQL Server: com.microsoft.sqlserver.jdbc.SQLServerDriver

Creating Database Connection

String url = "jdbc:mysql://localhost:3306/mydb"; String user = "root"; String password = "password"; Connection con = DriverManager.getConnection(url, user, password);

Statement Interface (Basic)

  • Execute static SQL statements
  • Vulnerable to SQL injection
  • Slower for repeated executions
  • Use for simple, one-time queries

Statement Usage

Statement stmt = con.createStatement(); String sql = "SELECT * FROM employees WHERE emp_id = 101"; ResultSet rs = stmt.executeQuery(sql); while(rs.next()) { System.out.println(rs.getString("name")); } rs.close(); stmt.close();

PreparedStatement Interface

  • Precompiled SQL statements with parameters
  • Uses placeholders (?) for dynamic values
  • Prevents SQL injection attacks
  • Better performance for repeated executions
  • RECOMMENDED over Statement

PreparedStatement Usage

String sql = "SELECT * FROM employees WHERE emp_id = ?"; PreparedStatement pstmt = con.prepareStatement(sql); pstmt.setInt(1, 101); ResultSet rs = pstmt.executeQuery(); while(rs.next()) { System.out.println(rs.getString("name")); }

CallableStatement Interface

  • Execute stored procedures in the database
  • Supports IN, OUT, and INOUT parameters
  • Can return ResultSet or single values
  • Better performance for complex database logic

Sample Stored Procedure

CREATE PROCEDURE get_employee_salary( IN emp_id INT, OUT emp_salary DOUBLE ) BEGIN SELECT salary INTO emp_salary FROM employees WHERE emp_id = emp_id; END;

CallableStatement Usage

String sql = "{call get_employee_salary(?, ?)}"; CallableStatement cstmt = con.prepareCall(sql); cstmt.setInt(1, 101); cstmt.registerOutParameter(2, Types.DOUBLE); cstmt.execute(); double salary = cstmt.getDouble(2); System.out.println("Salary: " + salary); cstmt.close();

ResultSet Interface

  • Represents query result set
  • Contains metadata about columns
  • Cursor moves forward by default (TYPE_FORWARD_ONLY)
  • Can retrieve data by column index or name
  • Must close ResultSet explicitly

ResultSet Data Retrieval

  • getString(int/String) - Get String value
  • getInt(int/String) - Get Integer value
  • getDouble(int/String) - Get Double value
  • getDate(int/String) - Get Date value
  • getBoolean(int/String) - Get Boolean value

SELECT Statement

String sql = "SELECT * FROM employees WHERE salary > ?"; PreparedStatement pstmt = con.prepareStatement(sql); pstmt.setDouble(1, 50000); ResultSet rs = pstmt.executeQuery(); while(rs.next()) { int id = rs.getInt("emp_id"); String name = rs.getString("name"); double salary = rs.getDouble("salary"); } rs.close(); pstmt.close();

INSERT Statement

String sql = "INSERT INTO employees (name, salary) VALUES (?, ?)"; PreparedStatement pstmt = con.prepareStatement(sql); pstmt.setString(1, "John Doe"); pstmt.setDouble(2, 75000); int rows = pstmt.executeUpdate(); System.out.println(rows + " row(s) inserted"); pstmt.close();

UPDATE Statement

String sql = "UPDATE employees SET salary = ? WHERE emp_id = ?"; PreparedStatement pstmt = con.prepareStatement(sql); pstmt.setDouble(1, 80000); pstmt.setInt(2, 101); int rows = pstmt.executeUpdate(); System.out.println(rows + " row(s) updated"); pstmt.close();

DELETE Statement

String sql = "DELETE FROM employees WHERE emp_id = ?"; PreparedStatement pstmt = con.prepareStatement(sql); pstmt.setInt(1, 101); int rows = pstmt.executeUpdate(); System.out.println(rows + " row(s) deleted"); pstmt.close();

CRUD Operations

  • CREATE - INSERT statement, uses executeUpdate()
  • READ - SELECT statement, uses executeQuery()
  • UPDATE - UPDATE statement, uses executeUpdate()
  • DELETE - DELETE statement, uses executeUpdate()

Exception Handling in JDBC

  • JDBC operations throw SQLException
  • Always use try-catch for JDBC code
  • Always close resources in finally block
  • Use try-with-resources (Java 7+) for auto-closing

Try-Catch-Finally Pattern

try { pstmt = con.prepareStatement(sql); rs = pstmt.executeQuery(); } catch(SQLException e) { e.printStackTrace(); } finally { if(rs != null) rs.close(); if(pstmt != null) pstmt.close(); if(con != null) con.close(); }

Try-With-Resources (Java 7+)

try (PreparedStatement pstmt = con.prepareStatement(sql)) { pstmt.setInt(1, 101); ResultSet rs = pstmt.executeQuery(); while(rs.next()) { System.out.println(rs.getString("name")); } } catch(SQLException e) { e.printStackTrace(); }

Transaction Management

  • Transaction - Atomic unit of multiple database operations
  • All-or-nothing criteria: success or rollback
  • By default, autocommit is enabled (each statement is a transaction)
  • Supports ACID properties (Atomicity, Consistency, Isolation, Durability)

Disabling AutoCommit

con.setAutoCommit(false); // Disable autocommit // Execute multiple operations pstmt1.executeUpdate(); pstmt2.executeUpdate(); pstmt3.executeUpdate(); con.commit(); // Commit all or con.rollback(); // Rollback all

Commit and Rollback

  • commit() - Persist all changes to database
  • rollback() - Discard all changes (undo)
  • Called explicitly when autocommit is disabled
  • Always rollback on exception for data consistency

Transaction Example

try { con.setAutoCommit(false); pstmt1.setDouble(1, 5000); pstmt1.setInt(2, 102); pstmt1.executeUpdate(); // Deposit pstmt2.setDouble(1, 5000); pstmt2.setInt(2, 101); pstmt2.executeUpdate(); // Withdraw con.commit(); } catch(SQLException e) { con.rollback(); }

Transaction Isolation Levels

  • READ_UNCOMMITTED - Lowest isolation, dirty reads possible
  • READ_COMMITTED - Default, prevents dirty reads
  • REPEATABLE_READ - Prevents non-repeatable reads
  • SERIALIZABLE - Highest isolation, phantom reads prevented

Setting Isolation Level

con.setTransactionIsolation( Connection.TRANSACTION_READ_COMMITTED ); // Or con.setTransactionIsolation(2);

Batch Operations

  • Execute multiple SQL statements as a batch
  • Better performance than executing one by one
  • Reduces network round trips to database
  • Use addBatch() and executeBatch()

Batch Operations Example

String sql = "INSERT INTO employees (name, salary) VALUES (?, ?)"; PreparedStatement pstmt = con.prepareStatement(sql); for(int i = 0; i < 100; i++) { pstmt.setString(1, "Employee" + i); pstmt.setDouble(2, 50000); pstmt.addBatch(); } int[] results = pstmt.executeBatch(); System.out.println("Batch executed"); pstmt.close();

DatabaseMetaData

  • Provides information about the database
  • Get table names, column info, primary keys, etc.
  • Obtain via con.getMetaData()
  • Useful for dynamic query building and table exploration

DatabaseMetaData Usage

DatabaseMetaData dbmd = con.getMetaData(); System.out.println("DB: " + dbmd.getDatabaseProductName()); System.out.println("Version: " + dbmd.getDatabaseProductVersion()); ResultSet tables = dbmd.getTables(null, null, "%", new String[]{"TABLE"}); while(tables.next()) { System.out.println(tables.getString("TABLE_NAME")); }

Connection Pooling

  • Reuse database connections instead of creating new ones
  • Improves performance significantly
  • Use libraries like HikariCP, C3P0, DBCP
  • Manage pool size, timeout, max connections
  • Essential for production applications

Why Connection Pooling?

  • Creating connections is expensive (network I/O)
  • Pooling reduces connection creation overhead
  • Connections are borrowed and returned to pool
  • Multiple applications can share pool efficiently
  • Can set limits to prevent resource exhaustion

Best Practices (1/2)

  • Always use PreparedStatement over Statement
  • Always use try-with-resources for auto-closing
  • Handle SQLException explicitly
  • Disable autocommit for multi-step transactions
  • Always rollback on exception

Best Practices (2/2)

  • Use batch operations for bulk inserts
  • Use connection pooling in production
  • Prevent SQL injection with parameterized queries
  • Set appropriate isolation levels for concurrency
  • Keep transactions short and focused

Common Mistakes

  • ❌ Not closing ResultSet, Statement, Connection
  • ❌ Using String concatenation for SQL (SQL injection)
  • ❌ Not handling SQLException
  • ❌ Creating new connections for each operation
  • ❌ Long-running transactions blocking other queries
