What is HikariCP?

  • HikariCP - Ultra-reliable, high-performance JDBC connection pool
  • Lightweight and fast (smallest footprint)
  • Zero-overhead production-ready library
  • Excellent for microservices and cloud applications
  • Default connection pool for Spring Boot applications
  • Apache 2.0 licensed, open source

Why Connection Pooling?

  • Creating DB connections is expensive (network I/O, authentication)
  • Pooling reuses connections instead of creating new ones
  • Dramatically improves application performance
  • Reduces database server load and memory usage
  • Prevents connection exhaustion and resource leaks
  • Better transaction management and isolation

HikariCP Advantages

  • Fastest - Micro-optimized for speed
  • Reliable - Battle-tested in production
  • Lightweight - Only 128 KB JAR file
  • Simple - Minimal configuration needed
  • Well-maintained - Active development

Essential Configuration

  • maximumPoolSize - Max connections (default: 10)
  • minimumIdle - Min idle connections (default: 10)
  • connectionTimeout - Max wait time (default: 30s)
  • idleTimeout - Max idle time (default: 10min)
  • maxLifetime - Max connection age (default: 30min)

Create HikariConfig

HikariConfig config = new HikariConfig(); config.setJdbcUrl("jdbc:mysql://localhost:3306/mydb"); config.setUsername("root"); config.setPassword("password"); config.setMaximumPoolSize(10); config.setMinimumIdle(5); config.setConnectionTimeout(30000); config.setIdleTimeout(600000); config.setMaxLifetime(1800000);

Create HikariDataSource

HikariDataSource dataSource = new HikariDataSource(config); // Get connection from pool Connection conn = dataSource.getConnection(); // Use connection... // Return connection to pool conn.close(); // Close pool (application shutdown) dataSource.close();

public class EmployeeDAO { private static HikariDataSource dataSource; static { HikariConfig config = new HikariConfig(); config.setJdbcUrl("jdbc:mysql://localhost:3306/mydb"); config.setUsername("root"); config.setPassword("password"); config.setMaximumPoolSize(10); dataSource = new HikariDataSource(config); } public Connection getConnection() throws SQLException { return dataSource.getConnection(); } }

Add Employee (INSERT)

public void addEmployee(String name, double salary) throws SQLException { String sql = "INSERT INTO employees (name, salary) " + "VALUES (?, ?)"; try (Connection conn = getConnection(); PreparedStatement pstmt = conn.prepareStatement(sql)) { pstmt.setString(1, name); pstmt.setDouble(2, salary); pstmt.executeUpdate(); System.out.println("Employee added successfully"); } catch(SQLException e) { throw new SQLException("Insert failed", e); } }

Get All Employees (SELECT)

public List<Employee> getAllEmployees() throws SQLException { List<Employee> employees = new ArrayList<>(); String sql = "SELECT * FROM employees"; try (Connection conn = getConnection(); PreparedStatement pstmt = conn.prepareStatement(sql); ResultSet rs = pstmt.executeQuery()) { while(rs.next()) { Employee emp = new Employee( rs.getInt("emp_id"), rs.getString("name"), rs.getDouble("salary") ); employees.add(emp); } } return employees; }

Update Employee (UPDATE)

public void updateEmployee(int empId, double newSalary) throws SQLException { String sql = "UPDATE employees SET salary = ? " + "WHERE emp_id = ?"; try (Connection conn = getConnection(); PreparedStatement pstmt = conn.prepareStatement(sql)) { pstmt.setDouble(1, newSalary); pstmt.setInt(2, empId); int rows = pstmt.executeUpdate(); System.out.println(rows + " row(s) updated"); } }

Delete Employee (DELETE)

public void deleteEmployee(int empId) throws SQLException { String sql = "DELETE FROM employees WHERE emp_id = ?"; try (Connection conn = getConnection(); PreparedStatement pstmt = conn.prepareStatement(sql)) { pstmt.setInt(1, empId); int rows = pstmt.executeUpdate(); System.out.println(rows + " row(s) deleted"); } }

Bulk Insert with Batch

public void bulkAddEmployees(List<Employee> employees) throws SQLException { String sql = "INSERT INTO employees (name, salary) " + "VALUES (?, ?)"; try (Connection conn = getConnection(); PreparedStatement pstmt = conn.prepareStatement(sql)) { for(Employee emp : employees) { pstmt.setString(1, emp.getName()); pstmt.setDouble(2, emp.getSalary()); pstmt.addBatch(); } pstmt.executeBatch(); System.out.println("Batch insert completed"); } }

Transaction Handling

public void transferBonus(int fromId, int toId, double amount) throws SQLException { Connection conn = getConnection(); try { conn.setAutoCommit(false); deductBonus(conn, fromId, amount); creditBonus(conn, toId, amount); conn.commit(); System.out.println("Transfer completed"); } catch(SQLException e) { conn.rollback(); throw e; } finally { conn.close(); } }

Connection Health Checks

  • testOnBorrow - Validate before giving connection
  • testOnReturn - Validate when connection returned
  • testWhileIdle - Test idle connections periodically
  • validationQuery - SQL to test connection health
  • Default: minimal validation (SELECT 1 for MySQL)

Configure Validation

config.setConnectionTestQuery("SELECT 1"); config.setLeakDetectionThreshold(60000); config.setValidationTimeout(5000); // Spring Boot spring.datasource.hikari.connection-test-query=SELECT 1 spring.datasource.hikari. leak-detection-threshold=60000

Leak Detection

  • Detects connections not returned to pool
  • Default threshold: 0 (disabled)
  • Set to reasonable value (e.g., 60000ms = 1 minute)
  • Logs warnings about suspicious connection holding
  • Helps identify resource leaks in application

HikariCP Best Practices

  • Create single DataSource (static, shared instance)
  • Use try-with-resources for connection handling
  • Set appropriate pool size for your workload
  • Enable leak detection in development
  • Monitor pool metrics in production

Pool Size Calculation

  • Formula: connections = (core_count × 2) + spare_connections
  • Core count from Runtime.getRuntime().availableProcessors()
  • Typical: 8-10 connections for most applications
  • Too small: connection timeout errors
  • Too large: wastes memory and database resources

Monitor Pool Health

  • Total connections in pool
  • Active (in-use) connections
  • Idle connections available
  • Connection wait time
  • Connection creation rate

Common Issues

  • ❌ Pool size too small → connection timeouts
  • ❌ Not closing connections → connection leak
  • ❌ Creating multiple DataSources → resource waste
  • ❌ Ignoring idle timeout → stale connections
  • ❌ Very long maxLifetime → connection death

Understanding Timeouts

// connectionTimeout (30s default) // Time to wait for available connection // idleTimeout (10min default) // Max idle time before connection closed // maxLifetime (30min default) // Max age of connection in pool // validationTimeout (5s default) // Time for validation query
